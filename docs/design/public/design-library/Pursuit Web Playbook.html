Pursuit · Web Adaptation Playbook

The same product, grown up to the desktop.

Pursuit is a mobile-first product, and the mobile UI is the source of truth. This playbook is the system for taking that exact identity — the lavender-blush canvas, the deluge purple, the category-themed event surfaces — onto tablet and desktop. It constrains and reflows. It never restyles. The goal is a Pursuit that feels like a cultural guide on every screen, not a marketing site bolted onto an app.

Mobile = source of truth Tablet 768–1199 Desktop 1200+ 3 adaptation levels Shared tokens, web layout layer

Reflow, don't redesign

Same components, rearranged for the cursor and the viewport. If you're restyling, you've gone too far.

Constrain, don't stretch

Content lives inside a max width. Cards keep their proportions. Nothing balloons edge-to-edge.

Theme from the category

Event surfaces are coloured by their category variant — never by the platform. Web inherits this wholesale.

One source of truth

Colour, radii, shadows, type and voice are shared tokens. Only layout and display sizing get a web layer.

What the current port gets wrong

The first web port stretches cards across the full window, drops content into edge-to-edge gutters, labels nav with raw route names (index, group_plans), and falls back to a system serif because the webfonts never load. Every fix in this playbook traces back to one of those four mistakes.

01

Responsive Layout System

A page shell that holds the product still.

Mobile stays exactly as shipped. Above it, two breakpoints add structure rather than density: a tablet that introduces the two-up grid and a persistent top bar, and a desktop that centres everything inside a fixed content column with generous gutters. The single most important rule on the web is the max width — it's what stops Pursuit from feeling like a dashboard.

Content max-width

Body content caps at 1200px, centred. The window can be 2560px wide; the reading experience never is.

Gutters

clamp(20px, 5vw, 64px). They breathe open on desktop and tuck to the mobile 20–24px at the low end.

Grid

1 → 2 → 3 columns. Card gutter grows 16 → 22 → 26px. Columns are capped, so cards never widen past their sweet spot.

Section rhythm

Mobile's 40px section gap opens to 56–64px on desktop. More air, same intervals.

02

Typography Adaptation

Don't redesign the type. Scale the display, anchor the text.

The type system is shared: Work Sans for headings and UI, Poppins for body, the same weights and tracking on every platform. What changes on the web is one thing — large display headings get bigger to fill the extra space, while reading text and UI labels stay put. Bigger body copy on a wide screen reads as clumsy, not premium.

The rule

Only display roles scale across breakpoints. Body, labels, captions and card titles are fixed. Reading text never exceeds ~18px and never runs wider than ~68 characters.

Reading width

On mobile the screen does the constraining for you. On desktop you have to do it yourself: prose, descriptions and curator notes get a hard max-width so lines stay short and scannable.

Flag — webfonts

The current port renders headings in a system serif because Work Sans never loads. Ship the fonts (or keep the Google Fonts link) before anything else — half of the "off-brand" feeling in the screenshots is just the missing typeface.

03

Navigation System

A lightweight editorial top bar — never a SaaS sidebar.

Mobile keeps its bottom tab bar, untouched. The web earns a persistent top bar: the Pursuit wordmark on the left, a few quiet text links, a search field, and an avatar on the right. It reads like a magazine masthead — clean, low-contrast, content-first. Crucially, the links use human labels (Discover, Plans, Group, Profile), not the route names the current port leaked to the surface.

Desktop

Persistent 64px bar, white surface, hairline silverSand base. Wordmark in deluge; links in leather, active in deluge.

Tablet

Same bar, condensed: text links stay, the search field collapses to an icon, the bottom tab bar drops. The cursor makes bottom tabs redundant.

Mobile

Unchanged. Bottom tab bar (Discover · Plans · Group · Profile), deluge active accent — exactly as the app ships.

Search

Inline pill field on desktop. Focus draws a 3px deluge ring. It's a convenience, not the centrepiece.

04

Card System

The cards already work. Cap them and let them breathe.

This is the part to protect. The mobile event cards are well-proportioned, so the web's only job is to stop them distorting. Every card gets a min and max width, a fixed image ratio, and a calm hover. They sit in the capped grid from §01, so a card on a 27-inch monitor is the same shape as a card on an iPad.

05

Interaction System

Subtle and premium. Nothing flashy.

The web adds a layer mobile never had: the hover. Used well it makes the product feel responsive and considered; overused it feels like a toy. Pursuit's interactions are short (140–220ms), eased, and physical — a small lift, a deepening shadow, a colour settling in. Everything here respects prefers-reduced-motion.

06

Screen Classification Framework

Three levels. Decide once, build with confidence.

Not every screen needs a bespoke web build, and not every screen survives on mobile markup alone. Before writing a line of web code, classify the screen into one of three levels. The level tells you exactly how much divergence is allowed — and stops the team from over-building simple screens or under-building complex ones.

Level 1

Shared completely

One component, one layout, both platforms. Just centred and width-capped on the web. Reach for this whenever a screen is a single, content-driven column.

// same .tsx, max-width wrapper

Level 2

Shared components, web layout

Identical building blocks — cards, badges, stat tiles, CTA — rearranged by a .web.tsx layout. The pieces don't change; their composition does.

// Screen.web.tsx + shared parts

Level 3

Separate web implementation

The desktop experience is structurally different — multi-pane, persistent chrome, parallel content. Build it for the web, still themed from shared tokens.

// dedicated web screen

6.4

Classifying any future screen

1
Is it a single content column?

Yes → Level 1. Centre it, cap the width, ship. No → question 2.

2
Can the same components be rearranged to fill the width?

Yes → Level 2. Write a .web.tsx layout over the shared parts. No → question 3.

3
Does desktop need structurally different chrome or parallel content?

Yes → Level 3. Build a dedicated web screen — still themed from shared tokens. No → you're probably at Level 2; don't over-build.

Rule of thumb

Promote a screen to a higher level only when its arrangement must change — never because something merely needs to be bigger. Size is a token; structure is a level.

07

Token Strategy

One core. A thin web layer on top.

The token system should make divergence the exception. Identity tokens — colour, radii, shadow, category variants, type families, motion — are shared and platform-agnostic. Only the things that are genuinely a function of viewport — spacing macros, display sizing, layout dimensions — get a web override that sits on top of the shared base.

Token groupStrategyWhy
ColoursSharedThe brand is the palette. Deluge, prim, thunder and the 8 category variants are identical everywhere.
RadiiSharedA 12px card is a 12px card. Corner language carries the brand.
ShadowsSharedSame elevation ramp; the web simply uses the hover end of it more.
Category variantsSharedAccent + tint pairs theme event surfaces by category, never by platform.
Type families & weightsSharedWork Sans + Poppins + Courier, same weights. Only display sizes diverge.
MotionSharedDurations and easings are universal; the web just has more surfaces (hover) to apply them to.
Spacing macrosWeb layerThe 4px base unit is shared, but screen padding and section gaps grow on larger viewports.
Display type sizingWeb layerHero and heading sizes scale up; the base scale stays shared.
Layout dimensionsWeb onlyMax-widths, gutters, grid columns, nav height, card min/max — concepts mobile doesn't have.

7.2

The web layout layer

The web file doesn't redefine the brand — it only adds the concepts mobile never needed, and reuses the shared spacing scale for everything else.

tokens/layout/layout.web.ts 
// Web-only layout layer. Inherits the shared 4px spacing scale.
import { space } from "../core/typography.base";

export const layout = {
  maxWidth:     1200,                         // content column cap
  gutter:       "clamp(20px, 5vw, 64px)",      // outer breathing room
  readingWidth: 680,                          // max measure for prose
  navHeight:    { tablet: 58, desktop: 64 },

  grid: {
    columns: { mobile: 1, tablet: 2, desktop: 3 },
    gutter:  { mobile: space.base, tablet: 22, desktop: 26 },
  },

  card: {
    standard: { min: 280, max: 390, ratio: "16 / 10" },
    rec:      { min: 240, max: 320 },
  },

  breakpoints: { tablet: 768, desktop: 1200 },
} as const;
tokens/typography/type.web.ts 
// Only display roles scale. Body / labels inherit the shared base.
import { type } from "../core/typography.base";

export const display = {
  hero:    "clamp(30px, 6vw, 48px)",   // 30 → 48
  h2:      "clamp(24px, 5vw, 40px)",   // greeting
  h3:      "clamp(20px, 3vw, 27px)",   // section header
  body:    type.base,                  // 16 — unchanged
  label:   type.label,                 // 14 — unchanged
};
08

Future-Proofing

So the next screen takes minutes to classify, not meetings.

The point of a playbook is that it keeps working after this document is closed. These are the durable principles — pin them above the team's desk. When a new Pursuit screen needs a web build, it should be answerable by these alone.

Mobile is the source of truth

Start from the shipped mobile screen every time. The web is an adaptation of it, never a fresh canvas.

Reflow, don't redesign

If you're changing colours, type or component styling for the web, stop. You should only be changing arrangement.

Constrain, don't stretch

Everything lives in a max-width. Cards and reading measures are capped. Width becomes whitespace, not bigger boxes.

Theme from the category

Event surfaces colour themselves from their category variant. Platform never enters the theming decision.

Size is a token, structure is a level

Bigger? Adjust a token. Differently arranged? That's a level promotion. Don't confuse the two.

Editorial, not enterprise

When in doubt, look like a magazine, not a dashboard. No dense sidebars, no data slop, no gradient soup.

Subtle motion only

Hover and focus are quiet and fast. If an interaction draws attention to itself, it's wrong.

Accessible by default

Every interactive element has a visible focus state and respects reduced motion. The current port has neither.

8.1

New-screen checklist

✓

Classified into Level 1 / 2 / 3 using the three questions in §6.4.

✓

Content capped at the layout max-width; prose held to the reading measure.

✓

Components reused from the shared kit — no web-only restyling of shared parts.

✓

Nav uses the top bar (desktop/tablet) and the bottom tabs (mobile), with human labels.

✓

Cards sit in the capped grid with min/max widths — nothing balloons.

✓

States defined for hover, focus and active; motion within the shared tokens.

✓

Category theming driven by the variant, if it's an event surface.

✓

Tokens pulled from the shared core; only layout/display from the web layer.

The one-line test

If a Pursuit user moved from their phone to their laptop mid-session, they should feel they opened the same app on a bigger screen — not that they landed on a different website. That feeling is the whole brief.
